# Canvas Brief — Project Log

## The Idea
A tool that connects to Stanford Canvas, pulls course module content (cases, readings, study questions), and uses Claude to generate printable study briefs for each class session.

---

## Phase 1: Local Python Script

**Goal:** Fetch Canvas data and generate briefs on my laptop.

### v1 — Assignment-based (first attempt)
- Built `canvas_brief.py` connecting to the Canvas API with a personal access token
- Fetched active courses and upcoming assignments
- Sent assignment descriptions to Claude API to generate summaries
- Saved output as markdown files in a `briefs/` folder

### v2 — Modules instead of assignments
- Shifted from assignments to **Modules**, since that's where professors put session content (cases, readings, study questions)
- Added course filtering: ignored "24F-MBA-Program", "Career Management - MBA'26", and "OB-374 / 375 Prequalification"
- Added skipping for non-session modules like "Canvas Help for Students" and "Course Overview"
- Rewrote the Claude prompt to summarize cases, answer study questions, and summarize multiple readings separately

### v3 — PDF downloads
- Added `pypdf` to download and extract text from PDF files attached in Canvas modules
- This was key — many professors upload case studies and readings as PDFs, not as Canvas pages
- Non-PDF files and external links are flagged but skipped

### v4 — Per-session PDFs with incremental generation
- Changed output from one brief per course to **one PDF per session** (e.g., "Session 1 - April 3: Weddington Way")
- Added `fpdf2` for PDF generation from markdown
- Hit Unicode issues (em dashes, smart quotes) with fpdf2's built-in fonts — solved with a `sanitize_text()` function and an HTML-based rendering approach using `write_html()`
- Added a `.generated.json` tracker so re-running the script only generates briefs for **new sessions** (useful as professors upload content week-by-week)
- First full run: **48 PDFs** generated across 6 courses

## Phase 2: Web App

**Goal:** Make this accessible to classmates — anyone at Stanford should be able to connect their Canvas, generate briefs, and download them. Briefs are shared so one person generating helps everyone.

### Architecture decisions
- **Framework:** Flask (same language as the CLI script, beginner-friendly)
- **Database:** SQLite (simple, no external service needed)
- **Auth:** Canvas API token paste (no Stanford SSO — keeps it simple)
- **Sharing:** All generated PDFs are public in a shared library
- **Cost model:** I fund the Claude API costs
- **Metrics:** Track usage for a potential case study

### App structure
```
canvas-brief/
├── app.py                 ← Flask web app (routes, views)
├── canvas_api.py          ← Canvas API helpers (extracted from CLI script)
├── brief_generator.py     ← Claude API + PDF creation (extracted from CLI script)
├── database.py            ← SQLite DB + analytics queries
├── canvas_brief.py        ← Original CLI script (still works)
├── gunicorn.conf.py       ← Server config for Render
├── Procfile               ← Tells Render how to start the app
├── templates/
│   ├── base.html          ← Shared layout (nav, footer, flash messages)
│   ├── index.html         ← Landing page with token input + library link
│   ├── dashboard.html     ← User's courses with generate buttons per session
│   ├── library.html       ← Public browseable library of all briefs
│   └── metrics.html       ← Analytics dashboard
├── static/style.css       ← Stanford cardinal-themed CSS
└── briefs/                ← Generated PDF storage
```

### Pages
- `/` — Landing: paste Canvas token or browse existing briefs
- `/dashboard` — Shows your courses and sessions with "Generate Brief" buttons
- `/library` — Public library of all generated briefs, downloadable by anyone
- `/metrics` — Usage analytics (unique users, downloads, generates, top briefs)
- `/api/metrics` — JSON endpoint for programmatic access to metrics
- `/health` — Diagnostic endpoint showing system status

### Key features
- **Shared library:** A brief generated by one student is available to all students in the same course
- **Incremental:** Only generates briefs for sessions that don't already exist in the database
- **Token security:** Canvas tokens are stored in the session cookie only, never persisted to disk
- **Metrics tracking:** Every login, generate, and download is logged with timestamps and anonymized IPs

## Phase 3: Deployment to Render

**Goal:** Get the app live on the internet.

### Setup
- Pushed code to GitHub (github.com/jenoodle94/canvas-brief)
- Created a web service on Render connected to the repo
- Added `ANTHROPIC_API_KEY` as an environment variable on Render

### Issues encountered and resolved

**1. Model not found (404)**
- Initial model ID `claude-sonnet-4-6-20250514` didn't exist on the account
- Listed available models via the API, switched to `claude-sonnet-4-6`

**2. Worker killed — "Perhaps out of memory?" (SIGKILL)**
- First appeared on Render's free tier (512MB RAM)
- Reduced workers from 2 to 1, added memory caps on PDF processing
- Turned out to be a **timeout issue, not OOM** — gunicorn's default 30s timeout was killing workers mid-request because Canvas + Claude calls take 30-60s

**3. Gunicorn timeout not taking effect**
- `render.yaml` settings weren't used because the service was created manually on Render's dashboard
- Solution: added `gunicorn.conf.py` which gunicorn auto-detects regardless of how the service was started
- Set timeout to 300 seconds

**4. 502 Bad Gateway after adding gunicorn.conf.py**
- Hardcoded port 10000 in the config, but Render assigns port via `$PORT` env var
- Fixed by reading `os.getenv('PORT', '10000')`

**5. Upgraded to Render Pro**
- Free tier's 512MB RAM was marginal
- Pro tier (2GB RAM) provides comfortable headroom for PDF processing + Claude calls

### Debugging tools added along the way
- `/health` endpoint — shows status of data directory, database, and Anthropic API
- Logging throughout the generate endpoint
- Frontend shows actual error messages instead of generic "Server error"
- Global Flask 500 error handler returns JSON with traceback for API routes
